# UI MVP 第二批面板修改记录 2026-06-18

## 一、本轮修改范围

在 `UI_Setting` 试点通过后，本轮继续改造两个低风险按钮型面板：

- `Assets/Scripts/UI/Boot/UI_MainMenu.cs`
- `Assets/Scripts/UI/UI_DeathPanel.cs`

新增 Presenter：

- `Assets/Scripts/UI/Boot/UI_MainMenuPresenter.cs`
- `Assets/Scripts/UI/UI_DeathPanelPresenter.cs`

补充中文注释：

- `Assets/Scripts/Interface/IUIView.cs`
- `Assets/Scripts/UI/UIPresenterBase.cs`

本轮不修改 UI 外观，不移动 prefab，不修改场景，不重命名原有 Button 绑定的方法名。

## 二、问题位置、成因、后果

### 1. UI_MainMenu 直接处理启动流程和游戏流程

问题位置：

```csharp
// Assets/Scripts/UI/Boot/UI_MainMenu.cs
void Start()
{
    transform.root.GetComponentInChildren<UI_Setting>(true).LoadAudioSetting();
    transform.root.GetComponentInChildren<UI_FadeScreen>().FadeIn();
    AudioManager.Instance.StartBGM("playList_MainMenu");
}

public void PlayGame()
{
    AudioManager.Instance.PlayGlobalSFX("btn_click");
    GameManager.Instance.ContinuePlay();
}
```

问题成因：

- 主菜单 View 同时负责找其他 UI 组件、播放 BGM、播放按钮音效、切换游戏流程。
- 场景流程和音频流程散落在 UI View 中。

可能后果：

- 后续主菜单如果增加“新游戏、继续游戏、设置、存档选择”等功能，`UI_MainMenu` 会快速膨胀。
- UI 显示代码和游戏流程耦合，测试和排查困难。

详细修改方案：

- 新增 `UI_MainMenuPresenter`。
- `UI_MainMenu` 实现 `IUIView`。
- `UI_MainMenu.Start()` 只调用 `presenter.Initialize()`。
- `PlayGame()`、`QuitGame()` 保持原方法名，只转发给 Presenter。
- `LoadSettingPanelAudio()`、`PlayFadeIn()` 保留在 View，作为显示相关方法供 Presenter 调用。

为何这样修改：

- 保留原 Button 绑定方法名，避免破坏 `MainMenu.unity`。
- Presenter 统一处理主菜单流程，View 只显示和转发输入。
- 空引用保护降低 UI 子节点缺失时的崩溃风险。

### 2. UI_DeathPanel 直接写死场景名并调用 GameManager

问题位置：

```csharp
// Assets/Scripts/UI/UI_DeathPanel.cs
public void GoToTown()
{
    GameManager.Instance.ChangeScene("Level_0", RespawnType.NonSpecific);
}

public void GoToMainMenu()
{
    GameManager.Instance.ChangeScene("MainMenu", RespawnType.NonSpecific);
}
```

问题成因：

- 死亡面板 View 直接处理场景切换。
- 魔法字符串 `"Level_0"`、`"MainMenu"` 写在 View 层。

可能后果：

- 场景名变更时需要到 UI View 中查找和修改。
- 后续如果死亡复活流程加入扣钱、播放动画、二次确认、统计数据等逻辑，View 会继续膨胀。

详细修改方案：

- 新增 `UI_DeathPanelPresenter`。
- `UI_DeathPanel` 实现 `IUIView`。
- `GoToTown()`、`GoToCheckPoint()`、`GoToMainMenu()` 保持原方法名，只转发给 Presenter。
- 场景名常量集中到 Presenter 中。

为何这样修改：

- 死亡面板是典型按钮型 UI，适合用简化版 MVP。
- 保留 public 方法名可以避免 `Canvas.prefab` 中 Button 绑定断裂。
- Presenter 集中管理流程，View 更容易维护。

### 3. MVP 基础类注释不清晰

问题位置：

```csharp
// Assets/Scripts/UI/UIPresenterBase.cs
public abstract class UIPresenterBase<TView> where TView : class, IUIView
```

问题成因：

- 基础类刚建立时较轻量，注释不足。
- 当前要继续扩展更多面板，需要更明确的中文规则。

可能后果：

- 后续新增面板时，容易重新把业务逻辑写回 View。
- 初学者不容易判断哪些代码应该放 View，哪些应该放 Presenter。

详细修改方案：

- 给 `IUIView` 增加中文职责说明。
- 给 `UIPresenterBase<TView>` 增加中文职责说明和 `Attach/Detach` 注释。

为何这样修改：

- 本项目是学习型项目，清晰注释比复杂抽象更有价值。
- 统一规则可以让后续 UI 重构按相同模式推进。

public abstract class UIPresenterBase<TView> where TView : class, IUIView

## 四、本轮建立和强化的规则

View 层职责：

- 持有 Unity 序列化字段和场景节点引用。
- 接收 Unity 生命周期和 Button/Slider/Toggle 事件。
- 调用 Presenter，不直接处理游戏流程。
- 可以提供少量显示方法，例如播放淡入、刷新显示。

Presenter 层职责：

- 管理按钮背后的业务流程。
- 访问 Manager、场景名、音效名等业务依赖。
- 管理魔法字符串常量。
- 不挂在 prefab 上，不直接控制 UI 外观。

## 五、验证方式

建议在 Unity Editor 中验证：

1. 打开 `MainMenu.unity`，确认无编译错误。
2. 进入主菜单，确认设置音量加载、淡入、主菜单 BGM 正常。
3. 点击 Play，确认按钮音效播放，并继续进入游戏。
4. 点击 Quit，编辑器中不会真正退出构建程序，但 Console 不应报错。
5. 在游戏中触发死亡面板，分别点击回城、回检查点、回主菜单，确认场景流程正常。
6. 检查 `Canvas.prefab` 和 `MainMenu.unity` 的 Button 绑定没有 Missing Method。

## 六、风险说明

- 本轮保留所有原有 public 方法名，降低 prefab/scene 绑定断裂风险。
- 新增 Presenter 是普通 C# 类，不需要挂载到 GameObject 上，因此不需要修改 prefab。
- `UI_MainMenu` 中查找 `UI_Setting` 和 `UI_FadeScreen` 的逻辑仍在 View 中，但已加空引用保护；后续如果引入 UI Root/Context，可以继续减少查找。
- 当前仍保留 `GameManager.Instance`、`AudioManager.Instance` 单例访问，先保证行为稳定，后续再考虑更彻底的依赖隔离。
