## 二、问题位置、成因、后果

### 1. UI_Setting 同时承担 View、业务逻辑、数据读写、系统跳转

问题位置：

```csharp
// Assets/Scripts/UI/UI_Setting.cs
public class UI_Setting : MonoBehaviour
{
    private Player player;
    [SerializeField] private Toggle toggle_HealthBar;
    [SerializeField] private AudioMixer audioMixer;
    ...
}
```

问题成因：

- 初期开发时把 UI 显示、输入响应、设置存档、音频 Mixer 修改、玩家血条控制、返回主菜单流程全部写在一个 MonoBehaviour 中。
- 面板代码直接依赖 `Player`、`PlayerPrefsDataMgr`、`SaveManager`、`GameManager`、`AudioMixer`，职责边界不清晰。

可能后果：

- 后续新增设置项时，`UI_Setting` 会越来越像 God Object。
- UI 外观调整和业务逻辑修改互相影响，调试成本增加。
- 其他面板难以复用“View 只显示、Presenter 处理逻辑”的规则。

详细修改方案：

- 新增 `IUIView` 作为 View 标记接口。
- 新增 `UIPresenterBase<TView>` 作为轻量 Presenter 基类。
- 新增 `UI_SettingPresenter`，负责设置读取、设置保存、音量应用、血条开关、返回主菜单。
- 保留 `UI_Setting` 上原有 UnityEvent 可调用方法，避免 prefab 按钮/Slider/Toggle 绑定丢失。

为何这样修改：

- 这是最小 MVP 试点，不需要一次性重构全部 UI。
- `UI_Setting` 面板逻辑简单、列表少、依赖清晰，适合作为第一块样板。
- 保留 Inspector 方法名可以降低 Unity 序列化引用断裂风险。

修改后：


Presenter 中：

```csharp
private void ApplyMixerVolume(string parameter, float value)
{
    if (!HasView || View.AudioMixer == null || string.IsNullOrEmpty(parameter))
        return;

    float safeValue = Mathf.Max(value, MinMixerSliderValue);
    float mixerValue = Mathf.Log10(safeValue) * View.MixMultiplier;
    View.AudioMixer.SetFloat(parameter, mixerValue);
}
```

## 四、本轮建立的规则

### View 规则

`UI_Setting` 作为 View，只负责：

- 持有 Unity 序列化字段，例如 Toggle、Slider、AudioMixer 参数。
- 接收 UnityEvent、生命周期事件。
- 把输入转发给 Presenter。
- 提供必要的显示方法，例如 `SetAudioSliderValues`。

### Presenter 规则

`UI_SettingPresenter` 负责：

- 读取和保存设置数据。
- 处理 Slider 值到 AudioMixer 分贝值的转换。
- 控制玩家血条显示隐藏。
- 执行返回主菜单流程。
- 做空引用保护和输入值保护。

### 暂不处理内容

- 暂不接入 Frame 的 `UIMgr`，避免当前 UI 根节点和 prefab 流程一次性大改。
- 暂不移动 `SettingData`，避免扩大本轮变更范围。后续可以迁移到 Data/Model 文件夹。
- 暂不重构 `UI_MainMenu`、`UI_DeathPanel`、`UI_CanvasRoot`，等待本试点稳定后再复制模式。

## 五、验证方式

建议在 Unity Editor 中验证：

1. 打开 `MainMenu.unity`，确认项目无编译错误。
2. 进入设置面板，拖动 BGM/SFX Slider，确认音量变化正常。
3. 关闭设置面板，再打开，确认 Slider 值会保存并恢复。
4. 进入游戏场景后打开设置面板，切换血条 Toggle，确认玩家血条显示/隐藏正常。
5. 点击返回主菜单，确认会保存游戏并切换到 `MainMenu`。
6. 检查 Console 是否出现空引用异常。

## 六、风险说明

- `UI_Setting` 保留了原有 public 方法名，Inspector 绑定理论上不会断。
- `FindAnyObjectByType<Player>()` 仍存在，但已被限制在 Presenter 中；后续可以通过依赖注入或 GameContext 进一步替换。
- Slider 值为 0 时，现在会使用 `0.0001f` 防止 `Mathf.Log10(0)` 产生负无穷，行为上更稳定。
