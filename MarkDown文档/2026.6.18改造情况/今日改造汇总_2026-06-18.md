# 今日改造汇总 2026-06-18

## 一、总体目标

今天的改造主要围绕两个方向推进：

- 先修复会影响稳定性的 P0 问题。
- 再逐步建立 UI 的 MVP 改造规则，让 View 只负责显示和输入转发，Presenter 负责业务流程。

本日改造遵守了以下约束：

- 不修改 UI 外观。
- 不主动移动 prefab。
- 保留原有 public 方法名，尽量避免 Unity Inspector/Button 绑定丢失。
- 每轮修改都新增 Markdown 记录，包含问题、成因、后果、方案和验证方式。

## 七、当前遗留设计点

### 1. Slot 中仍有 FindAnyObjectByType

当前 `UI_InventoryItemSlot.Awake()` 中仍有：

```csharp
this.playerInventory = FindAnyObjectByType<Inventory_Player>();
```

这个设计不是最终理想方案。

今天暂时保留的原因：

- 第 4 次改造目标是先迁移普通 Slot 的业务交互逻辑。
- 如果同时改依赖注入，会牵动 `UI_ItemSlotGroup`、`UI_Inventory`、`UI_Storage`、`UI_Merchant`、`UI_InGame`。
- 为了小步可验证，先不扩大改动面。

后续建议：

- 做一次 `Slot 依赖注入化` 小改造。
- 给 Slot 增加 `Initialize(Inventory_Player playerInventory, UI_CanvasRoot canvasRoot)`。
- 由上层面板或 SlotGroup 统一给所有 Slot 注入依赖。
- 逐步移除 Slot 内部的 `FindAnyObjectByType`。

### 2. 仓库、商人、快捷栏 Slot 仍未完全 MVP 化

后续可以按以下顺序继续：

1. `UI_InventoryEquipSlot`
2. `UI_StorageItemSlot`
3. `UI_MerchantSlot`
4. `UI_QuickItemSlot`
5. `UI_QuickItemSlotOption`

建议仍然一次只改一小组，改完就在 Unity 中验证对应交互。
