# UI 物品 Slot MVP 第 4 次改造记录 2026-06-18

## 一、本轮修改范围

本轮目标是开始“物品 Slot 交互 MVP 化”，但不一次性重构全部背包、仓库、商人和快捷栏逻辑。

优先试点：

- `Assets/Scripts/UI/InventoryAndStorage/UI_InventoryItemSlot.cs`

新增 Presenter：

- `Assets/Scripts/UI/InventoryAndStorage/UI_InventoryItemSlotPresenter.cs`
- `Assets/Scripts/UI/InventoryAndStorage/UI_InventoryItemSlotPresenter.cs.meta`

暂不改造：

- `UI_InventoryEquipSlot`
- `UI_StorageItemSlot`
- `UI_MerchantSlot`
- `UI_QuickItemSlot`
- `UI_QuickItemSlotOption`

原因：这些子类涉及装备卸下、仓库转移、商人买卖、快捷栏绑定等更具体的业务规则。本轮先稳定普通背包 Slot 的 MVP 规则，再逐步迁移子类。

## 二、问题位置、成因、后果

### 1. UI_InventoryItemSlot 同时负责显示和业务交互

问题位置：

```csharp
// Assets/Scripts/UI/InventoryAndStorage/UI_InventoryItemSlot.cs
public virtual void OnPointerDown(PointerEventData eventData)
{
    if (eventData.button == PointerEventData.InputButton.Right)
        return;

    if (this.currentItemInSlot == null)
    {
        this.canvasRoot.itemToolTip.ShowToolTip(false);
        return;
    }

    if (this.currentItemInSlot.itemData.itemType == ItemType.Material)
        return;

    bool alternativeInput = Input.GetKey(KeyCode.LeftControl);
    if (alternativeInput)
    {
        this.playerInventory.RemoveAllStack(this.currentItemInSlot);
    }
    else
    {
        if (this.currentItemInSlot.itemData.itemType == ItemType.Consumable)
        {
            this.playerInventory.TryToUseItem(this.currentItemInSlot);
        }
        else
        {
            this.playerInventory.TryToEquipItem(this.currentItemInSlot);
        }
    }
}
```

问题成因：

- Slot View 既负责图标和数量显示，又直接处理点击后使用物品、装备物品、整组移除物品。
- View 直接访问 `Inventory_Player`、`UI_CanvasRoot`、`Input.GetKey`、Tooltip。

可能后果：

- 后续仓库、商人、快捷栏复用 Slot 时，容易在子类里继续堆业务逻辑。
- 普通背包格子的显示变化和业务规则变化互相影响。
- 点击规则修改时需要在 View 里处理大量分支，维护成本高。

详细修改方案：

- 新增 `UI_InventoryItemSlotPresenter`。
- `UI_InventoryItemSlot` 保留显示逻辑：`UpdateSlot`、图标、数量、当前物品引用。
- `UI_InventoryItemSlot` 的 `OnPointerDown`、`OnPointerEnter`、`OnPointerExit` 改为转发给 Presenter。
- Presenter 负责判断左键/右键、空物品、材料、Ctrl 整组移除、消耗品使用、装备、Tooltip 显示和隐藏。

为何这样修改：

- 这是物品 Slot MVP 化的最小安全切入点。
- 保留 `UpdateSlot` 和原有事件方法名，避免 prefab 和继承链受影响。
- 子类仍可继续 override 原方法；后续可以逐个子类迁移到自己的 Presenter。

{

Presenter 中：

```csharp
public virtual void HandlePointerDown(PointerEventData eventData)
{
    if (!HasView || eventData.button == PointerEventData.InputButton.Right)
        return;

    Inventory_Item item = View.CurrentItemInSlot;
    if (item == null || item.itemData == null)
    {
        HideItemToolTip();
        return;
    }

    if (item.itemData.itemType == ItemType.Material)
        return;

    bool transferFullStack = Input.GetKey(KeyCode.LeftControl);
    if (transferFullStack)
    {
        playerInventory?.RemoveAllStack(item);
        return;
    }

    if (item.itemData.itemType == ItemType.Consumable)
    {
        playerInventory?.TryToUseItem(item);
        return;
    }

    playerInventory?.TryToEquipItem(item);
}
```

### 3. Presenter 创建规则

修改后：

```csharp
protected virtual UI_InventoryItemSlotPresenter CreatePresenter()
{
    return new UI_InventoryItemSlotPresenter(this.playerInventory, this.canvasRoot);
}

protected void EnsurePresenter()
{
    if (this.presenter != null)
        return;

    this.presenter = CreatePresenter();
    this.presenter.Attach(this);
}
```

## 四、本轮建立和强化的规则

`UI_InventoryItemSlot` 作为 View：

- 持有图标、数量文本、当前物品引用。
- 接收鼠标点击、进入、离开事件。
- 把输入转发给 Presenter。
- 不直接决定物品使用、装备、整组移除。

`UI_InventoryItemSlotPresenter` 作为 Presenter：

- 判断点击类型。
- 判断是否为空物品、材料、消耗品、装备。
- 调用 `Inventory_Player` 执行使用、装备、移除。
- 控制 Tooltip 显示和隐藏。

## 五、风险说明

- 本轮保留 `OnPointerDown`、`OnPointerEnter`、`OnPointerExit`、`UpdateSlot` 方法，不破坏 Unity EventSystem 调用。
- 本轮不改子类逻辑，避免一次牵动仓库、商人、快捷栏。
- `UI_MerchantSlot` 中调用 `base.OnPointerDown(eventData)` 的行为仍然保留，只是 base 内部改为转发给 Presenter。
- `UI_InventoryEquipSlot`、`UI_StorageItemSlot`、`UI_QuickItemSlot` 等子类 override 后仍按原逻辑运行。

## 六、验证方式

建议在 Unity Editor 中验证：

1. 打开项目，确认无编译错误。
2. 打开背包，鼠标悬停普通物品格，确认 Tooltip 显示正常。
3. 鼠标离开普通物品格，确认 Tooltip 隐藏正常。
4. 左键点击消耗品，确认仍能使用。
5. 左键点击装备，确认仍能装备。
6. 左键点击材料，确认不触发使用或装备。
7. 按住 LeftControl 点击普通物品，确认整组移除逻辑仍按原行为执行。
8. 打开商人界面，确认玩家物品格左键仍能复用普通背包点击逻辑。
