## 1. 真实入口场景确认

### 源代码或配置位置

- `Assets/Scenes/MainMenu.unity`
- `ProjectSettings/EditorBuildSettings.asset`

### 问题成因

项目中同时存在正式入口场景 `MainMenu.unity` 和测试场景 `SampleScene.unity`。如果 Build Settings 中没有把 `MainMenu.unity` 作为正式入口，打包后可能进入测试场景，或代码调用 `GameManager.ChangeScene("MainMenu", ...)` 时找不到场景。

### 造成的后果

- 打包后无法从死亡面板、设置面板返回主菜单。
- 点击继续游戏或主菜单流程可能失败。
- 测试场景可能被误作为正式入口。

### 本轮处理

用户已确认真实入口是 `MainMenu.unity`，并已自行修改。  
本轮没有修改 Build Settings 或场景配置。

### 为何这样处理

用户已经处理该项，本轮只记录风险和验证方式，避免重复改动项目配置。

### Unity Editor 验证方式

1. 打开 `File > Build Profiles` 或 `Build Settings`。
2. 确认第一个启用场景是 `Assets/Scenes/MainMenu.unity`。
3. 进入 Play Mode，确认从主菜单可以继续游戏。
4. 在游戏内测试死亡返回主菜单、设置返回主菜单。

## 2. Frame 中 `TextUtil.cs` 疑似语法破损

### 源代码位置

- `Assets/Scripts/Frame/Util/TextUtil.cs`

### 问题成因

命令行读取该文件时出现明显中文乱码，部分字符串在终端显示为疑似引号破损。但用户反馈 Unity 可以成功编译，说明当前 Unity 使用的编码解析结果可能没有实际语法错误，或终端显示与 Unity 编译器读取结果不一致。

### 造成的后果

如果它实际存在语法破损，会造成 Unity 脚本编译失败，整个项目无法进入 Play Mode。  
如果只是终端显示乱码，则暂时不会影响运行，但会影响后续维护和审查。

### 本轮处理

本轮没有修改 `TextUtil.cs`。

### 详细修改方案

无代码修改。后续如果 Unity Console 出现该文件的编译错误，再单独处理：

- 优先确认文件真实编码。
- 只修复语法破损的字符串和注释。
- 不在本轮无依据地重写该工具类，避免引入新风险。

### 为何这样处理

用户明确表示 Unity 能成功编译。当前没有可靠证据证明 `TextUtil.cs` 在 Unity 环境中实际报错，因此本轮不修改它，避免把“终端显示乱码”误当作“源文件必然损坏”。

### Unity Editor 验证方式

1. 打开 Unity Editor。
2. 等待脚本编译完成。
3. 确认 Console 中没有 `TextUtil.cs` 的语法错误。
4. 如果后续使用 `TextUtil`，再针对具体 API 做功能验证。

## 3. Runtime 代码直接引用 `UnityEditor`

### 源代码位置

- `Assets/Scripts/Data/Item/ItemData_SO.cs`
- `Assets/Scripts/Data/Item/ItemListData_SO.cs`
- `Assets/Scripts/Object/Object_CheckPoint.cs`

### 问题成因

`ItemData_SO.cs` 和 `ItemListData_SO.cs` 顶部直接 `using UnityEditor;`，虽然实际调用 `AssetDatabase`、`EditorUtility` 的代码在 `#if UNITY_EDITOR` 内，但 `using UnityEditor;` 本身仍位于运行时代码区域。

`Object_CheckPoint.cs` 中的 `UnityEditor.EditorUtility.SetDirty(this)` 已经在 `#if UNITY_EDITOR` 内，且没有文件顶部 `using UnityEditor;`，本轮不需要修改。

### 造成的后果

在 Player Build 中，`UnityEditor` 命名空间不可用。运行时脚本顶部直接 `using UnityEditor;` 可能导致打包编译失败。

### 详细修改方案

将 `using UnityEditor;` 包进 `#if UNITY_EDITOR`：

```csharp
#if UNITY_EDITOR
using UnityEditor;
#endif
```

保留原有 `OnValidate` 和编辑器工具逻辑，不改变道具 ID 生成、道具数据库自动收集等行为。

### 本轮实际修改

- `ItemData_SO.cs`
  - 将 `using UnityEditor;` 包入 `#if UNITY_EDITOR`。
- `ItemListData_SO.cs`
  - 将 `using UnityEditor;` 包入 `#if UNITY_EDITOR`。
- `Object_CheckPoint.cs`
  - 未修改，因为它的 Editor API 调用已经被 `#if UNITY_EDITOR` 包住。

### 为何这样修改

这是最小安全修改：

- 不移动脚本。
- 不拆分 Editor 文件。
- 不改变 ScriptableObject 数据逻辑。
- 只解决 Player Build 中 `UnityEditor` 命名空间不可用的问题。

### Unity Editor 验证方式

1. 等待 Unity 自动重新编译。
2. 确认 Console 无编译错误。
3. 执行一次目标平台 Build，确认不再因 `UnityEditor` 引用失败。
4. 在 Inspector 修改 `ItemData_SO`，确认 `saveID` 仍能在编辑器内正常生成。

## 4. 输入事件重复订阅风险

### 源代码位置

- `Assets/Scripts/FiniteStateMachine/Character/Player/Player.cs`
- `Assets/Scripts/UI/UI_CanvasRoot.cs`

### 问题成因

原代码在 `Player.OnEnable()` 和 `UI_CanvasRoot.SetUpInputUIControl()` 中使用 lambda 表达式直接订阅 Input System 事件：

```csharp
action.performed += ctx => ...
```

这类匿名 lambda 后续无法用相同引用退订。`Player.OnDisable()` 只是 `p_Input.Disable()`，没有解除事件订阅。若对象被禁用再启用，或重新绑定输入，就可能叠加回调。

### 造成的后果

- 同一次按键触发多次技能。
- 同一次快捷键使用多次物品。
- UI 面板打开/关闭多次切换。
- 对象被事件引用，增加内存泄漏和 MissingReference 风险。

### 详细修改方案

#### `Player.cs`

- 新增命名回调方法：
  - `OnMovementPerformed`
  - `OnMovementCanceled`
  - `OnSpellPerformed`
  - `OnMousePerformed`
  - `OnInteractPerformed`
  - `OnQuickItem1Performed`
  - `OnQuickItem2Performed`
- 新增：
  - `BindInputActions()`
  - `UnbindInputActions()`
- `OnEnable()` 中先 `UnbindInputActions()`，再 `BindInputActions()`，最后启用输入。
- `OnDisable()` 中先退订，再禁用输入。

#### `UI_CanvasRoot.cs`

- 将 UI 输入 lambda 改为命名回调：
  - `OnSkillTreeUIPerformed`
  - `OnInventoryUIPerformed`
  - `OnAlternativePerformed`
  - `OnAlternativeCanceled`
  - `OnSettingUIPerformed`
- 新增 `UnbindInputUIControl()`。
- `SetUpInputUIControl()` 绑定新输入前，先解绑旧输入。
- `OnDestroy()` 中解绑 UI 输入。
- 增加 `p_InputSet` 空值保护。
- 修正 `UI_CanvasRoot.Instance` 未赋值的问题，保持原本意图。

### 为何这样修改

命名方法可以被可靠退订。  
在绑定前先解绑，可以避免重复调用 `SetUpInputUIControl()` 或异常生命周期导致的重复订阅。  
这属于生命周期安全修复，不改变玩家输入映射和 UI 快捷键行为。

### Unity Editor 验证方式

1. 进入游戏后按移动、交互、技能、快捷栏，确认行为正常。
2. 反复打开/关闭背包和技能树，确认每次只切换一次。
3. 进入设置面板，再返回游戏，确认玩家输入恢复正常。
4. 切换场景后再次测试技能和 UI 输入，确认没有重复触发。
5. 死亡后显示死亡面板，确认输入禁用正常。

## 5. 快捷栏空引用风险

### 源代码位置

- `Assets/Scripts/InventoySystem/Inventory_Player.cs`

### 问题成因

`quickItems` 是长度为 2 的数组，但初始元素可能为 `null`。原代码在 `TryToUseQuickItem()` 中直接访问：

```csharp
item.itemData
```

如果快捷栏为空，就会触发 `NullReferenceException`。

同时 `TryToUseItem()` 假设传入物品一定存在且一定是可用消耗品。如果传入 `null`、非消耗品、无效果数据的物品，也可能空引用。

### 造成的后果

- 玩家未设置快捷物品时按快捷键，游戏报错。
- 快捷栏物品数据异常时中断逻辑。
- UI 或输入层后续重构时更容易触发隐藏空引用。

### 详细修改方案

- `TryToUseQuickItem(int slotNumber)`
  - 增加槽位范围检查。
  - 增加 `item == null || item.itemData == null` 检查。
- `TryToUseItem(Inventory_Item itemToUse)`
  - 增加传入物品空值检查。
  - 查找消耗品时跳过空物品和空 `itemData`。
  - 增加 `cosumableItem == null || cosumableItem.itemEffectData == null` 检查。

### 为何这样修改

这是最小安全保护：

- 不改变快捷栏数据结构。
- 不改变物品使用成功时的行为。
- 只在无效输入或空槽位时提前返回，避免报错。

### Unity Editor 验证方式

1. 新游戏或清空快捷栏。
2. 不设置快捷物品，直接按快捷键 1 和 2。
3. Console 不应出现 NullReference。
4. 设置可用消耗品到快捷栏，按快捷键后仍能正常使用。
5. 快捷物品消耗完后，快捷栏替换或清空逻辑仍正常。
